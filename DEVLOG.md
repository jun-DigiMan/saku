# サクピタ 開発ログ

## プロダクト概要

**サクピタ**（Saku-Pita）は、営業チームが顧客との商談日程を効率よく調整・予約できるWebアプリ。
Googleカレンダーと連携し、チームメンバー全員の空き時間をリアルタイムで確認しながら、顧客情報の入力・確認メール送信・スプレッドシート記録までワンストップで完結する。

- **URL**: https://jun-digiman.github.io/sakupita/
- **リポジトリ**: https://github.com/jun-DigiMan/sakupita
- **技術スタック**: HTML / CSS / JavaScript（Vanilla JS・フレームワークなし）
- **外部API**: Google Calendar API / Gmail API / Google Sheets API / Google Drive API
- **認証**: Google Identity Services (GIS) OAuth 2.0
- **ホスティング**: GitHub Pages（無料・常時稼働・`git push`で自動反映）

---

## 開発の経緯・目的

- 既存の日程調整ツール（Calendly等）は外部向けで、社内複数メンバーのカレンダーを一括確認しながら調整する機能がない
- 顧客への確認メール送信・スプレッドシート記録など、商談調整に必要な作業を1つのツールに集約したかった
- Googleアカウントだけで完結するシンプルな設計にこだわった
- 外部SaaSに依存せず、自社で永続的に運用できるツールを目指した
- サーバーレス（静的ファイルのみ）で運用コストゼロ

## 実装機能・こだわりポイント 全詳細

### 🔐 認証・セッション管理

**実装内容：**
- Google Identity Services (GIS) の `requestAccessToken` でOAuth 2.0認証
- アクセストークンを有効期限付きで `localStorage`（`sakupita_token`）に保存
- リロード時に `gapi.client.setToken()` でトークンを即時復元 → サインイン画面を経由しない
- トークン期限の**55分前**にサイレント自動更新タイマーをセット → タブを開いている限り再認証不要（実質24時間以上維持）
- トークン切れ時は**再接続オーバーレイ**を表示。カレンダー表示を維持したまま「Googleカレンダーに接続する」ボタンで復帰可能
- サインアウト時は `sakupita_token` / `sakupita_signed_in` / `sakupita_busy_cache` / `sakupita_sheet_id` を全クリアしてサインイン画面へ

**こだわり：**
- リロードのたびにサインイン画面が出る問題を解消することに注力した。原因は `requestAccessToken({ prompt: '' })` が失敗してもコールバックを呼ばないという仕様で、最終的にトークン自体をlocalStorageに保存・復元する方式に落ち着いた
- サインアウトボタンはヘッダー右端にSVGアイコンで配置。メンバー管理（左）・サインアウト（右）の順番はUXを考慮して決定
- デモモードは開発初期に存在したが、本番運用に不要なため削除

### 📅 カレンダー表示

**実装内容：**
- 月〜日の7列グリッド × 時間帯別スロット表示（週表示）
- 前週・次週ナビゲーションボタン
- Googleカレンダー freebusy APIで全メンバーの予約済み時間帯を取得し、空き/予約済みをリアルタイム表示
- **busyDataキャッシュ**（`sakupita_busy_cache`）: 週単位・メンバー別に2時間TTLでキャッシュ。同じ週を再表示する際はAPIを呼ばず即表示

**こだわり：**

**当日ハイライトのデザイン（複数回試行）**
- 最初は青色 → 「ベース色の紺と被って見づらい」
- 緑色 → 「浮いて見える」
- オレンジ → 「土日カラーと混同しやすい」
- 水色の丸 → 「土曜日カラーっぽい」
- 最終決定: **紺色の丸に白文字**。メインカラーを崩さず、視認性が最も高い。曜日名も紺色で統一

**土日・祝日の視認性：**
- 土曜=青系、日曜=赤系でカラー分け
- 日本の祝日をハードコードで保持し、祝日スロットを視覚的に区別・予約不可に設定（誤予約防止）

### 🎨 ブランディング・デザイン

**実装内容：**
- オリジナルロゴ（サクピタ）を作成し複数箇所に配置：
  - ヘッダー左（白抜き版）
  - ファビコン
  - 確認メール本文
  - 予約完了画面
- ロゴはCanvasで生成しBase64エンコード → 外部ファイル参照なしのインライン表示
- 白抜き版・通常版の2種類を用意し、背景色に応じて使い分け
- メインカラー: 紺色（`--color-navy: #1a3353`）で統一。清潔感・信頼感を重視
- ナビゲーションバー: 紺色ヘッダーにロゴ・メンバー管理・サインアウトを配置

### 👥 メンバー管理

**実装内容：**
- メンバー追加: 名前 + Gmailアドレスで登録
- メンバー削除: 「削除」ボタンで即時削除
- 「連携済み」「未連携」バッジで状態を色分け表示
- メンバーごとにカレンダー表示用カラーを自動割り当て（重複なし）
- メンバー情報をlocalStorageに永続保存
- `calendar.app.google` URLを貼り付けてメンバー追加を試みる機能（corsproxy.io → allorigins.win → codetabs.com の3プロキシを順次試行）
  - ※ Firebase Dynamic Linksの仕様上、プロキシ経由でのメール自動取得は不可。手動入力フォールバックあり

### 📝 予約フォーム

**入力項目：**

| フィールド | 必須 | バリデーション | プレースホルダー例 |
|----------|------|------------|----------------|
| 会社名 | ✅ | 空欄チェック | 株式会社サクピタ |
| 担当者名 | ✅ | 空欄チェック + 自動スペース補完 | 山田 太郎 |
| メールアドレス | ✅ | 正規表現形式チェック | yamada@example.com |
| 電話番号 | ✅ | ハイフン形式チェック + 自動フォーマット | 03-1234-5678 |
| 部署 | ✅ | 空欄チェック | 営業部 |
| 役職 | ✅ | 空欄チェック | 部長 |
| コメント | — | なし | — |

**こだわり：**
- フォームを開いた瞬間に**会社名フィールドへ自動フォーカス** → 入力開始のストレスゼロ
- エラー時は該当フィールドを**赤枠ハイライト** + アラートで具体的なフォーマットを案内
- メールアドレスは形式チェックに加え、送信後の**バウンス検知**による二重確認
- 「処理中...」「メール到達確認中...」とボタンテキストをリアルタイムに更新し、処理状況を明示

### 📞 電話番号スマートフォーマット

**トリガー**: フィールドからフォーカスが外れた時（blur）

**変換仕様：**
```
入力例（全角・ハイフンなし）: ０９０１２３４５６７８
出力例: 090-1234-5678
```

| 桁数 | 種別 | フォーマット |
|-----|------|-----------|
| 11桁 | 携帯・IP電話 | XXX-XXXX-XXXX |
| 10桁 | 2桁市外局番（03/06） | XX-XXXX-XXXX |
| 10桁 | 3桁市外局番（011等） | XXX-XXX-XXXX |
| 10桁 | 4桁市外局番（0138等） | XXXX-XX-XXXX |
| 10桁 | 5桁市外局番（01372等） | XXXXX-X-XXXX |
| 10桁 | 特番（0120/0570/0800/0990） | XXXX-XXX-XXX |

**実装：**
- 全角数字・全角ハイフンを自動で半角変換
- **市外局番データベース**（`JP_AREA_CODES`）: 約200種類を収録。都道府県別に整理
- 5桁まで対応する最長一致アルゴリズム（`getAreaCodeLen()`）で地域ごとに最適なハイフン位置を自動判定
- ローカル番号は常に末尾4桁を固定し、残りを前半に配置

### 🏷 日本語名前自動分割

**トリガー**: フィールドからフォーカスが外れた時（blur）

**変換仕様：**
```
入力例（スペースなし）: 山田太郎
入力例（全角スペース）: 山田　太郎
出力例: 山田 太郎（半角スペース統一）
```

**実装：**
- `JP_SURNAMES` 辞書: 現在 **1,391種類**（随時拡充中・目標30万種）
- 5文字姓（勅使河原・武者小路）から1文字姓まで対応
- 沖縄系姓（比嘉・金城・玉城・知念・宮城・与那嶺・喜屋武等）も収録
- 最長一致アルゴリズムで正確に姓を判定（「長谷川」と「長谷」が両方ある場合は「長谷川」を優先）
- 辞書にない姓はフォールバックとして先頭2文字を姓と見なす

### 📧 メール送信・バウンス検知

**実装内容：**
- 予約完了時に**顧客**と**担当営業**の両方へ確認メールを自動送信（Gmail API）
- メール本文はHTMLメール。サクピタロゴ・Google Meet URL・商談日時を含む
- **バウンス自動検知フロー**:
  1. メール送信
  2. ボタンに「メール到達確認中...」と表示しながら**3秒待機**
  3. Gmail受信箱を `mailer-daemon` / `postmaster` + 宛先メールアドレスで検索
  4. バウンスメール検知 → アラート表示 + カレンダーに⚠️記録
  5. バウンスなし → ✅ 送信成功として記録

**カレンダーへの反映（PATCH）：**
- 送信成功: イベント説明欄に `✅ 確認メール送信済み / 送信先: xxx@xxx.com` を追記
- バウンス検知: イベントタイトルに `⚠️ メール未達 要アドレス確認 /` を前置き、説明欄にも記録
- GmailのAPI仕様上、存在しないアドレスでも送信時は200を返すため、バウンス検知は「送信後に跳ね返りメールが届いているか」で判定

## 技術的こだわりポイント

### ⚡ 高速化・軽量化

- **Discovery Docs廃止**: `gapi.client.init` での discovery docs 読み込みをやめ、`gapi.client.request()` 直接呼び出しに統一 → 初回ロード約600ms短縮
- **preconnect / dns-prefetch**: `<head>` に `apis.google.com` / `accounts.google.com` / `www.googleapis.com` への事前接続ヒントを追加
- **busyDataキャッシュ**: 週単位・メンバー別に2時間TTLでキャッシュ。ナビゲーション時のAPI再呼び出しを排除
- **フレームワークなし**: React/Vue等を不使用。Vanilla JSのみでバンドルサイズ最小化
- **トークン即時復元**: ページ読み込み時にlocalStorageからトークンを復元し、API呼び出しを即開始

### 🔄 リロード耐性（最重要課題）

リロードのたびにカレンダーが白紙になる問題の根本解決に最も時間をかけた。

**根本原因**: `requestAccessToken({ prompt: '' })` は失敗時にコールバックを呼ばない仕様のため、トークン取得失敗がサイレントになる

**最終解決策**:
1. 認証成功時にアクセストークンを有効期限付きでlocalStorageへ保存
2. `initGapiClient()` 実行時にlocalStorageからトークンを復元し `gapi.client.setToken()` で即設定
3. リロード後はAPI呼び出しが即可能 → カレンダーが即表示
4. トークン切れ時のみ `requestAccessToken({ prompt: '' })` でサイレント更新を試み、失敗時は再接続オーバーレイを表示

### 🛡 バリデーション設計

- フロントエンドのみで完結（サーバー不要）
- エラー時は該当フィールドを赤枠ハイライト + アラートで具体的な入力例を案内
- メールアドレス: 正規表現チェック（入力時）+ バウンス検知（送信後）の2段階確認
- 電話番号: ハイフン形式チェック（送信時）+ 自動フォーマット（blur時）の2段階
